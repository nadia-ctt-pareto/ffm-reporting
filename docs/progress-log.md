## 2026-07-21: Phase 7c — BYOK AI field polish (server-proxied, per-field)

**Summary**: Added per-field "Polish" buttons to the wizard's prose fields, powered by BYOK Anthropic key stored encrypted server-side under `AI_BYOK_ENCRYPTION_KEY`. The polish respects field-specific editorial intent (7 polishable fields, each with its own instructions); client is hard-excluded (dedupe key). Key never reaches the browser. Server-side RLS is owner-only with NO `is_admin()` branch (tighter than every other table). Per-user in-memory rate limiter (10 req/min, 2 concurrent max — honest caveat: per-Node-process, so multi-instance serverless doesn't hold the line; Redis upgrade is the documented path).

**What was built**:

- **Encryption** (`lib/server/ai-crypto.ts`): AES-256-GCM (`node:crypto`), one `encryptSecret`/`decryptSecret` pair over base64(iv + authTag + ciphertext). Decrypt failure degrades gracefully ("re-enter your key"), never a 500. Errors are message-only and generic (never embeds plaintext/ciphertext/raw crypto errors).
- **Key CRUD** (`lib/server/ai-keys.ts`): `getAiKeyStatus` (configured/hint/validatedAt/lastUsedAt, never plaintext), `setAiKey` (validates against Anthropic BEFORE encrypting, stores fingerprint), `deleteAiKey`, `getAiKeyPlaintext` (THE ONLY path that ever decrypts — plaintext lives in one call frame, used for one fetch, never assigned elsewhere or logged).
- **Polish server** (`lib/server/ai-polish.ts`, model: `claude-sonnet-5`): `validateAnthropicKey` (1-token ping pre-storage), `polishField` (rate-limit → decrypt → system/user prompts → Anthropic → clean result). Anthropic response body never logged (can echo back bad key's last chars). Errors map to marker tokens (anthropic_invalid_key / anthropic_rate_limited / anthropic_unavailable / anthropic_timeout / ai_key_unreadable), then curated by `curatedMessage` (lib/server/reports-service.ts).
- **Per-field registry** (`lib/prompts.ts`): `POLISH_FIELDS` (7 fields with distinct instructions) + `HOUSE_VOICE` (shared with Phase 8a's MCP Skill, so web app + Claude-via-MCP never diverge). `client` is hard-excluded (dedupe key — rewriting it breaks `(client, task)` and `(client, description)` dedup + project stamping).
- **UI** (`components/ai/PolishButton.tsx`): inline button on each polishable field. Suggestion below field (Accept / Discard / Undo). Original text never touched until Accept. Mid-flight edit discards stale suggestion (three guard points). `isAiPolishConfigured()` gates routes (404 if no Supabase or `AI_BYOK_ENCRYPTION_KEY` unset).
- **Routes** (`app/api/ai/key/route.ts`, `app/api/ai/polish/route.ts`): GET/PUT for key (never plaintext returned, fingerprint only), POST for polish (PolishRequest → polished text). Gated on `isAiPolishConfigured()`.
- **Settings section** (`components/settings/AiKeySection.tsx`): key upload (validated before storage), fingerprint + timestamps display, delete button, muted note if not configured.
- **Database** (`supabase/migrations/20260722000008_ai_keys.sql`): `ai_keys` table (user_id, key_ciphertext, key_hint, timestamps), `get_own_ai_key_ciphertext()` + `set_own_ai_key()` SECURITY DEFINER RPCs, owner-only RLS on all verbs (deliberately NO `is_admin()` — tighter than every other table), column-level grant excluding `key_ciphertext` from `authenticated`'s SELECT (RPC-only read access). Documented gotcha: ON CONFLICT ... DO UPDATE requires SELECT on EXCLUDED columns, so authenticated-role upsert is impossible — write through DEFINER RPC instead.

## 2026-07-21: Phase 8a — remote MCP server + Claude Skill (bearer-token auth)

**Summary**: Claude Code / Desktop / CLI can now connect to the app and read/write reports under the user's own ownership, enforced by the same RLS as the web UI. Ships the bearer-token path (8a); OAuth for claude.ai (8b) layers on top later with zero tool changes (OAuth token is just another api_tokens row). All privilege elevation is confined to ONE SQL function (`verify_api_token`) and ONE Node module (`lib/server/mcp-auth.ts`), the security core: bearer ffmcp_<token> → SECURITY DEFINER hash-lookup-revoked-expired-check + `last_used_at` stamp → user_id → HS256 JWT (signed server-only SUPABASE_JWT_SECRET, role: authenticated, NO app_metadata so is_admin() is false) → per-request Supabase client. That client runs the 8 MCP tools as the `authenticated` role under the IDENTICAL RLS as the web path — it does not re-implement ownership, it IS the RLS.

**What was built**:

- **Auth bridge** (`lib/server/mcp-auth.ts`): The ENTIRE privilege-elevation surface. `verifyApiToken` calls `verify_api_token` RPC via bare anon client. `mintMcpJwt` hand-rolled HS256 (no jose/jsonwebtoken dependency, dependency-light ethos). `userScopedClient` creates a fresh SupabaseClient per request with the JWT as its bearer. `bridgeMcpToken` chains them: token → RPC → JWT → scoped client. `verifyMcpAuth` is the `mcp-handler`'s `withMcpAuth` shape (`app/api/[transport]/route.ts` wires it in).

## 2026-07-21: Phase 7b (M1+M2+M3+M4) — HTTP repository, Supabase data plane, cross-machine sharing, local import

**Summary**: Connected the UI to Postgres via a new server layer and HTTP repository (`HttpReportsRepository`), enabling Supabase mode. The phase landed in four milestones: M1 (server plane with service functions and route handlers), M2 (cutover from localStorage to Postgres with failure resilience), M3 (cross-machine share links via tokens), M4 (one-time localStorage→Postgres import + consolidate defaults). Demo mode (no Supabase env) still uses localStorage. Two rounds of adversarial code review caught share-token leak (share_token readable by every authenticated user), read-schema DoS (over-length row poisoning all users' report lists), and missing SQL CHECK constraints.

**What was built**:

**M1 — Server plane**:
- **`lib/server/reports-service.ts`**: client-injected Supabase-client functions (never constructs a client, never service-role; RLS is correctness-critical) powering both HTTP handlers and Phase 8's eventual MCP tools. `ServiceError` with curated error codes; `mapPgError` routes unexpected DB errors → HTTP 500 (not 400); `curatedMessage` runs exactly once per error in `route-helpers.ts` to prevent double-curation flip-flopping conflict messages.
- **`lib/server/db-mapping.ts`**: snake_case row ↔ camelCase domain conversion; read-side timestamptz normalization (so `fmtDateShort` doesn't get `"T00:00:00+00:00"` and render NaN).
- **`lib/server/route-helpers.ts` + `request-guards.ts`**: curated error responses, CSRF guard (Sec-Fetch-Site allowlist + JSON content-type), streamed body-size limit, demo-mode misconfiguration warning.
- **`lib/schema/api.ts`**: transport schemas (separate from domain schemas; write boundaries have `.max()` caps matching future SQL CHECKs), kept out of the schema barrel so domain facades aren't polluted.
- **API routes** (`app/api/reports/route.ts`, `[id]/route.ts`, `[id]/share/route.ts`; `app/api/projects/route.ts`): thin handlers following pattern: config guard → auth check → Zod validate → service call → curated error.
- **`middleware.ts`**: unauthenticated `/api/*` returns 401 JSON (not 307 redirect to login).

**M2 — Cutover & failure resilience**:
- **`lib/data/http-reports-repository.ts`**: HTTP adapter behind the `getReportsRepository()` factory (localStorage stays as demo-mode fallback). Writes serialize through one queue to prevent Kanban-drag races where rapid drag-drop-drag-drop could each call merge-based `updateReportFields`, reading the pre-drag snapshot N times → last-write-won → all but one drag lost. `upsertMany()` is one POST → one transactional insert (not N fire-and-forget calls).

## 2026-07-18: Mobile P1-P7 (Responsive design) — 767px and 640px breakpoints, off-canvas nav drawer, touch-friendly Kanban

**Summary**: Made the app mobile-friendly at ≤767px primary breakpoint (≤1023px secondary, ≤640px for presentation nav). PMs can now consume/triage reports on phones; long-form authoring stays desktop. Every `@media` query is written `@media screen and (…)` — the `screen and` prefix is mandatory to prevent mobile rules from leaking into print.

**What was built**:

**Mobile navigation & layout** (mobile P1-P2):
- **Responsive spacing tokens** (`styles/theme.css`): `--page-pad-x`, `--page-pad-top`, `--page-pad-bottom`, `--header-pad-y`, `--row-action-pad-mobile` — overridden at `:root` inside `@media screen and (max-width: 767px)`. ~10 screen modules consume these instead of hardcoding pixel values. Centralizes the mobile scalar scaling.
- **Off-canvas drawer navigation** (`components/app/MobileNav.tsx`, `AppShell.tsx`, mobile P2): Below 768px, the desktop rail hides and a sticky mobile top bar (hamburger + brand) appears. Hamburger opens an off-canvas left drawer built from raw Radix `Dialog` parts (NOT the higher-level `components/ui/Dialog`, which is centered) that renders the same `Sidebar` verbatim — nav items never drift. `AppShell` owns `drawerOpen` state (client-only, never persisted); drawer auto-closes on nav-item click (via `Sidebar`'s `onNavigate` callback) and on `usePathname()` change (e.g. back button), plus via a `matchMedia('(min-width: 768px)')` listener if viewport resizes to desktop.
- **Sidebar props** (`showCollapseToggle`, `onNavigate`, `variant='rail'|'drawer'`): MobileNav passes these to hide collapse (meaningless in a drawer) and use `height: 100%` instead of `min-height: 100vh`.
- **Grid → block layout** (`AppShell.module.css`, mobile P2): `.shell` switches from `display: grid` (2 columns) to `display: block` at ≤767px, because grid items' containing block is the grid area (splits `min-height: 100vh` evenly, stretching short pages' header to ~50vp) and `position: sticky` on a grid item has zero travel. Under block flow, sticky gets real scroll travel. The media query repeats both `.shell` and `.shell.collapsed` selectors (media queries add no specificity; a bare rule would lose to the collapsed state's higher-specificity rule declared above, leaving a phantom column).
- **Drawer animation** (`MobileNav.module.css`, mobile P2): CSS `@keyframes` (not `transition`) because Radix's exit-animation detection checks `animationName` — a transition would report `'none'` and unmount synchronously. Guarded by `prefers-reduced-motion: reduce`.

**Mobile-friendly Tables** (mobile P3):
- **`stacked` opt-in** (`components/ui/Table.tsx`, `Table.module.css`): Card-stacking at ≤767px — header hidden, `.tr` → bordered card, `.td` shows label via `data-label::before`. Used by Dashboard, Daily, TaskList, Consolidate-sources (triage surfaces where row action + status must be visible). ## 2026-07-17: Phase 4 (Daily reports + weekly import) — daily report entity, shared wizard/report/present routes, weekly rollup aggregation, v1→v2 storage migration

**Summary**: Added a daily-report entity (one per day, covering all clients) with the same functionality as a weekly report — list, wizard, report screen, presentation deck — by generalizing the existing weekly components to a discriminated union (`WeeklyReport | DailyReport`) rather than building a parallel set. The weekly wizard gained an "Import This Week's Daily Reports" action that aggregates a week's dailies into the draft. Authored the v1→v2 localStorage migration (top correctness priority: **no user-entered data loss**) and the SQL delta migration.

**Data model** (`lib/types.ts`): `ReportCore` (shared fields) + `WeeklyReport`/`DailyReport` (discriminated by `kind`) + `type AnyReport = WeeklyReport | DailyReport`. `type Report = WeeklyReport` is kept as an alias so every Phase 1-3 call site (dashboard, weekly wizard, CSV, task/calendar views) compiles unchanged — sound because `ReportsRepository.getAll()` stays weeklies-only. `Draft` gained `kind` plus both `weekStart`/`weekEnd` and `date` unconditionally (unused pair is `''`); only `StepBasics` and `validateStep`'s step-1 branch on `kind`.

**Storage & migration** (`lib/data/local-storage-reports-repository.ts`): one unified key, `ff.reports.v2` (`AnyReport[]`), replacing the weeklies-only `ff.weekly-reports.v1`. The v1→v2 migration runs only when v2 is absent: reads v1 (if present), stamps `kind: 'weekly'` on every record, writes v2, and **leaves the v1 key in place forever as a backup** (never deleted). A corrupt v2 falls back to v1-recovery before reseeding — reseeding (which would discard v2's contents) only happens when neither a valid v2 nor a valid v1 payload exists. `lib/seed.ts` gained `seedDailyReports()` (5 daily reports, Mon-Fri 2026-07-13..17, the same week as weekly seed `r7`) with deliberately overlapping `(client, task)`/`(client, description)` pairs across days so the aggregator has real dedup scenarios out of the box.

**Aggregation** (`lib/aggregate.ts`): `aggregateDailiesIntoDraft(dailies, draft)` — a pure function reusing the exact `(client, task)`/`(client, description)`/`text` dedup predicates already in `useWizard`'s carry-forward Import panels. Tasks/risks dedupe with the LATEST day's version winning; priorities dedupe by exact text; touchpoints numerics sum (on top of the draft's own); the win is the latest non-empty daily win, but only when the draft's own win is still empty (never clobbers user-entered data).

**Repository generalization** (`lib/data/reports-repository.ts`): `getAll()` (weeklies only, unchanged), new `getAllDaily()`, `getById`/`upsert` over `AnyReport`, `update` patch typed `Partial<ReportCore>` (callers' richer per-kind patches are structurally assignable and still land at runtime via spread). `lib/hooks/useDailyReports.ts` mirrors `useReports.ts`'s optimistic-update pattern exactly.

**Shared wizard/report/present surfaces**: `useWizard`/`WizardScreen`/`WizardPage` take a `kind: 'weekly' | 'daily'` prop (default `'weekly'`) — only `StepBasics` (single Date field vs. Week Start/End + the weekly-only import panel) and `saveDraft`/`publish`'s output shape differ; steps 2-6 are untouched. `ReportScreen`/`ReportDeck`/`PresentScreen` generalized to `AnyReport`/`kind` the same way. `reportPeriodLabel`/`draftPeriodLabel` (`lib/report-utils.ts`) are the single place "weekly → `fmtWeekLabel`, daily → `fmtDateShort`" is decided, used by the list, wizard, report screen, deck, and `lib/csv.ts`'s `buildAllTasksCsv` (now `AnyReport[]`; CSV column `Week` → `Period`). `ShareDialog`'s `shareLinkFor` takes an optional `kind` param.

**One daily report per day, covering all clients** is enforced twice: in the wizard (`dailyDateConflict`/`validateStep`, checked on `next()`, `publish()`, AND `saveDraft()` — not just step validation, since Save Draft otherwise bypasses per-step checks) and in SQL (`supabase/migrations/20260717000002_daily_reports.sql`'s partial unique index `reports_one_daily_per_day on reports (report_date) where kind = 'daily'`, alongside a `kind` discriminant column, a nullable `report_date`, and a `reports_period_by_kind` CHECK constraint enforcing "exactly the period columns matching kind are set").

**New surfaces**: `/daily` (`DailyPage`/`DailyListScreen`, mirroring `DashboardPage`/`DashboardScreen` — Status filter, pagination, CSV export, "New Daily Report"), `/daily/new`, `/daily/[id]/edit`, `/daily/[id]`, `/daily/[id]/present` (outside the `(shell)` group, same pattern as the weekly present route).
