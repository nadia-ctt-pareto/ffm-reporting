## 2026-07-17: MVP scaffold — dashboard, 6-step wizard, dialogs, localStorage persistence

**Summary**: Imported the Claude Design prototype and ported it to Next.js 15 + React 19 + TypeScript behind a swappable `ReportsRepository` persistence layer.

**What was built**:
- Full-featured dashboard with stat cards, status/client/search filters, Sort By dropdown, reports table with status badges, View/Continue actions, empty state, and CSV export.
- 6-step new report wizard (Basics → Task Status → Touchpoints & Win → Risks & Blockers → Priorities → Review & Export) with per-step validation.

## 2026-07-17: Phase 1 (Foundation) — sidebar app shell, real routing, token-driven dark mode, Radix primitives, pagination, baseline schema

**Summary**: Refactored the single-page `WeeklyReportsApp` view-switcher into real App Router routes behind a sidebar app shell, replaced the scattered `darkMode ? {...} : {...}` inline-style branching with a token-driven dark theme (`data-theme` + semantic CSS custom properties), rebuilt `Dialog`/`Select`/`Switch` on Radix (`radix-ui`), added dashboard pagination, and authored the baseline Supabase schema + docs. Landed in three gate-green milestones (1a routing/shell, 1b dark mode/Radix, 1c pagination/migration/docs), each verified with `npm run build && npm run lint && npm run typecheck` plus a manual Playwright smoke pass.

**What was built**:
- **Routing (1a)**: `app/(shell)/layout.tsx` (`AppShell`: sidebar + content grid, collapse state persisted to `localStorage['ff.sidebar-collapsed']`), `app/(shell)/page.tsx` (`/`), `app/(shell)/reports/new/page.tsx`, `app/(shell)/reports/[id]/edit/page.tsx`. `WeeklyReportsApp.tsx` and the old `app/page.tsx` were deleted; their logic split into `components/dashboard/DashboardPage.tsx` and `components/wizard/WizardPage.tsx` (route-level orchestration, dialog hosting, `useReports()` per route). `Sidebar.tsx` hosts nav (Dashboard, active via `usePathname()`), the Dark Mode switch (moved out of the dashboard/wizard headers), and the collapse toggle, with a Radix `Tooltip` for collapsed-state nav labels.
- **Dark mode (1b)**: `components/theme/ThemeProvider.tsx` (context + `localStorage['ff.theme']` + mirrors `data-theme` onto `<html>`) plus a `next/script(beforeInteractive)` in `app/layout.tsx` for a no-FOUC pre-hydration flip. New `styles/theme.css` / `styles/theme-dark.css` centralize every semantic-token value the old inline `darkMode ? {...} : {...}` objects hardcoded. Every component that only took a `darkMode` prop for that branching (`DashboardScreen`, `WizardScreen`, `ImportPanel`, `StepTasks`, `StepRisks`, `StepPriorities`, `StatCard`) had it deleted; `rootStyle`/`filterBarStyle`/`lightPanelStyle`/`panelStyle` are gone. - **Baseline schema (1c)**: `supabase/migrations/20260717000001_initial_schema.sql` (`clients`, `reports`, `tasks`, `risks`, `priorities`, indexes, RLS-enabled with stub `authenticated_full_access` policies) + `docs/database-schema.md` (full TS-field ↔ column mapping tables + cutover checklist). No repository code reads this schema yet — it's versioned ahead of the actual Supabase cutover.

## 2026-07-17: Phase 3 (Views) — Task view (List/Kanban) + Calendar view (Week/Month), Radix Tabs/Popover, dnd-kit

**Summary**: Added two new sidebar views, both pure derivations over the existing `Report[]` data (no new storage): a Task view (`/tasks`) with List and drag-and-drop Kanban modes, and a Calendar view (`/calendar`) with Week and Month modes. Introduced `components/ui/Tabs.tsx` and `Popover.tsx` (Radix, unified `radix-ui` package) and `@dnd-kit/core`/`@dnd-kit/utilities` for the Kanban board. Daily reports (Phase 4) were explicitly not built, but the derivation selectors (`lib/view-utils.ts`, `lib/calendar.ts`) were written to extend cleanly to them.

**What was built**:
- **`lib/view-utils.ts`**: pure selectors — `allTasks(reports)` flattens every report's tasks into `{report, task}` pairs; `groupTasksByStatus(entries)` groups/sorts them by status in `TASK_STATUS_ORDER` (Blocked → In Progress → Complete), each group sorted by report `weekEnd` desc then task `deadline` asc (no persisted intra-column/group order); `reportsOverlappingRange(reports, startISO, endISO)` filters by ISO-string `localeCompare` overlap, written against a plain date range (not "week") so it extends to daily reports later.
- **`lib/calendar.ts`**: date math using `Date.UTC(y, m-1, d)` + `getUTCDay()`/`getUTCFullYear()`/`getUTCMonth()`/`getUTCDate()` **only** (never a local-time `new Date(isoString)`), matching the codebase's ISO-string convention — `addDaysISO`, `isoWeekday`, `startOfWeekISO`/`endOfWeekISO` (Monday-start), `addWeeksISO`, `firstOfMonthISO`/`addMonthsISO`, `monthGridDays` (Monday-start 6-row/42-cell grid, padded into neighboring months), `isSameMonth`, `monthLabel`, `shortDayLabel` (reuses `fmtDateShort`, doesn't reimplement month names).
- **`lib/report-utils.ts`**: added `withTaskStatus(report, taskId, status)` — a pure helper returning a new `tasks` array with one task's status changed; callers feed the result into the existing `useReports().updateReportFields` (optimistic update + fresh `updatedAt` + persist), so the Kanban drop path rides the exact same mutation path as every other edit in the app.
- **`components/ui/Tabs.tsx`**: Radix Tabs (`radix-ui`) styled as a square-cornered segmented control, `{value, onChange, items: {value, label, content}[]}` API (same `onChange(value)` convention as `Select`). Verified Radix leaves an inactive tab's `content` genuinely unmounted (`@radix-ui/react-tabs`'s `children: present && children`), so the Kanban board's `DndContext` never mounts while List mode is selected.
- **`components/ui/Popover.tsx`**: Radix Popover (`{trigger, children, align?}`), used by the Calendar month grid's "+N more" overflow disclosure.
- **`components/tasks/`**: `TaskViewScreen.tsx` (owns the List/Kanban `mode` toggle + derives `entries`/`grouped`), `TaskList.tsx` (grouped table, each row a `next/link` to `/reports/[id]`), `KanbanBoard.tsx` (one `@dnd-kit/core` `DndContext`: `PointerSensor` with `activationConstraint: {distance: 8}` + `KeyboardSensor`, `closestCorners` collision detection, `DragOverlay` for the floating card), `KanbanColumn.tsx` (`useDroppable` keyed by `TaskStatus`), `TaskCard.tsx` (`useDraggable` keyed by the composite `${reportId}::${taskId}` id via `taskCardId.ts`, since a task's status lives on its parent report).
- **`components/calendar/`**: `CalendarScreen.tsx` (owns the Week/Month `mode` toggle plus two independent anchors — `weekStart` always Monday-anchored, `monthStart` always 1st-of-month-anchored — so switching tabs never loses your place in the other view; Prev/Next/Today act on whichever anchor is active), `WeekGrid.tsx` (single Mon–Sun row, reports as spanning bars clipped to the week, one row per overlapping report), `MonthGrid.tsx` (Monday-start 6-row grid; real interval-lane packing per row — non-overlapping bars share a lane, overlapping ones stack; bars beyond `MAX_VISIBLE_LANES` (2) collapse into a "+N more" `Popover`).
- **`app/(shell)/tasks/page.tsx`** and **`app/(shell)/calendar/page.tsx`**: thin route wrappers matching `app/(shell)/page.tsx`'s pattern exactly — own `useReports()`, render nothing until `reports !== null` (no localStorage-during-SSR, no hydration mismatch). All List/Kanban and Week/Month state lives in the Screen components themselves (same rationale as `ReportScreen` owning its Share-dialog state directly) rather than a dedicated `*Page.tsx` orchestrator, since neither route has filter/pagination state to justify one.
- **`components/app/Sidebar.tsx`**: added "Tasks" (`/tasks`) and "Calendar" (`/calendar`) nav items alongside "Dashboard", using the same `usePathname()` active-state pattern.
- **Dependency**: `@dnd-kit/core@^6.3` + `@dnd-kit/utilities` — installed with **zero peer-dependency issues** against React 19/Next 15 (no `--legacy-peer-deps`, no `overrides` needed), same clean result as Radix in Phase 1.

**Design decisions worth flagging**:
- Click-vs-drag on a Kanban card is resolved entirely by `PointerSensor`'s 8px activation distance (a plain click never moves the pointer that far, so dnd-kit never starts a drag and the card's own `onClick` fires normally) plus dnd-kit's own built-in swallowing of the trailing `click` after a real drag (a one-shot document listener that stops propagation) — no custom click/drag disambiguation code was needed.
- `KeyboardSensor`'s default codes make a focused card's Enter key **start a drag** (its `start` codes are `[Space, Enter]`), not navigate — documented in `CLAUDE.md` rather than worked around, since overriding it would fight dnd-kit's own accessibility defaults.
- The Month grid's "+N more" overflow was verified against a genuinely overlapping scenario by temporarily injecting two overlapping-week draft reports into `localStorage` at runtime (via the running app's own persistence, not by editing `lib/seed.ts`) — the seed data's 7 reports never naturally overlap the same calendar week, so this was the only way to drive real overflow in the running app.
